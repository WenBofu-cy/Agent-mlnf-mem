# ag-mem-13-画像槽长期未活跃提醒单元 接口规格

---

## 基本信息

| 项 | 内容 |
|----|------|
| 模块编号 | ag-mem-13 |
| 模块名称 | 画像槽长期未活跃提醒单元 |
| 所属分区 | 二、漏斗一：用户画像漏斗 |
| 核心职责 | 定期扫描漏斗一中所有长期槽的最后活跃时间戳，检测超过90天未活跃的画像槽。对于符合条件的槽位，通过 ECC 社会心智模块（ag-ecc-10）向用户发出“是否保留该画像数据”的提醒通知。若用户确认保留，重置活跃时间戳；若用户选择删除或超时未响应，向 ag-mem-02 发起槽位清除请求。不参与任何认知决策，仅负责未活跃检测与提醒触发 |
| 依赖模块 | ag-mem-02（漏斗一专属调度单元，查询所有长期槽状态、发起清除请求）、ag-mem-06（画像槽数据隔离管控单元，获取槽位元数据读取权限） |
| 被依赖模块 | ag-ecc-10（社会心智模块，接收并展示用户提醒通知）、ag-mem-02（接收清除请求）、ag-mem-51（记忆变更日志追溯单元，记录提醒与清除事件） |


## 内部状态定义

| 状态 | 标识 | 含义 | 触发条件 |
|------|------|------|----------|
| 空闲等待 | `IDLE` | 无扫描任务，等待定时触发 | 系统初始化完成，等待扫描周期 |
| 扫描进行中 | `SCANNING` | 正在遍历长期槽列表，检查最后活跃时间 | 定时触发或收到手动扫描指令 |
| 等待用户响应 | `AWAITING_USER` | 已向用户发送提醒通知，等待用户决策 | 检测到超90天未活跃的长期槽 |
| 处理用户决策 | `PROCESSING_DECISION` | 用户已响应，正在执行保留或清除操作 | 收到用户反馈或响应超时 |
| 暂停服务 | `SYSTEM_PAUSED` | 系统紧急熔断 | 收到紧急熔断指令 |


## 输入数据

| 输入项 | 数据类型 | 来源模块 | 触发条件 | 优先级 |
|--------|----------|----------|----------|:---:|
| 扫描触发信号 | Struct（触发类型：定时/手动 + 扫描范围 + 未活跃阈值天数） | 内部定时器 或 ag-mem-01 总控漏斗F0 | 定时（每24小时）或收到手动扫描指令 | 普通 |
| 长期槽状态列表 | List（槽位编号 + 用户ID + 槽位创建时间 + 最后活跃时间 + 存储使用量） | ag-mem-02 漏斗一专属调度单元 | 响应扫描查询请求时 | **高** |
| 用户决策响应 | Struct（槽位编号 + 用户决策：保留/删除 + 响应时间戳） | ag-ecc-10 社会心智模块（通过 MemoryBus） | 用户在提醒通知中做出选择时 | **高** |
| 全局调度指令 | Enum（暂停/恢复/熔断） | ag-mem-01 总控漏斗F0 | 模式切换或紧急事件时 | **紧急** |


## 输出数据

| 输出项 | 数据类型 | 目标模块 | 输出条件 | 优先级 |
|--------|----------|----------|----------|:---:|
| 未活跃槽位提醒通知 | Struct（通知ID + 槽位编号 + 用户ID + 未活跃天数 + 最后活跃时间 + 建议操作选项） | ag-ecc-10 社会心智模块（通过 MemoryBus） | 检测到超90天未活跃的长期槽时 | 普通 |
| 槽位清除请求 | Struct（目标槽位编号 + 清除原因 + 清除策略 + 用户确认标记） | ag-mem-02 漏斗一专属调度单元 | 用户选择删除或响应超时 | **高** |
| 槽位活跃时间重置请求 | Struct（槽位编号 + 新活跃时间戳 + 重置原因） | ag-mem-02 漏斗一专属调度单元 | 用户选择保留时 | **高** |
| 扫描结果上报 | Struct（扫描时间 + 扫描槽位总数 + 未活跃槽位数 + 已发送提醒数 + 用户响应统计） | ag-mem-01 总控漏斗F0 | 每次扫描周期完成后 | 普通 |
| 提醒事件日志 | Struct（事件类型 + 槽位编号 + 用户决策 + 时间戳） | ag-mem-51 记忆变更日志追溯单元 | 每次用户响应或超时时 | 普通 |


## 未活跃判定与提醒策略

### 一、判定规则

| 条件 | 判定结果 | 动作 |
|------|:---:|------|
| 当前时间 - 最后活跃时间 > 90天 | 长期未活跃 | 发送提醒通知 |
| 当前时间 - 最后活跃时间 > 180天 | 超长期未活跃 | 发送提醒通知（标记为紧急） |
| 当前时间 - 最后活跃时间 ≤ 90天 | 正常活跃 | 跳过，不处理 |

### 二、提醒通知内容模板

```
通知类型：画像数据保留提醒
用户ID：{user_id}
未活跃天数：{inactive_days}
最后活跃时间：{last_active_date}
存储使用量：{storage_used}
建议选项：
  1. [保留] - 继续保留我的偏好与历史数据
  2. [删除] - 删除我的所有画像数据（不可恢复）
  3. [忽略] - 暂不处理，30天后再次提醒
默认处理：若7天内未收到用户响应，自动执行保留操作（重置活跃时间戳）
```

### 三、用户响应处理

| 用户选择 | 处理方式 |
|----------|------|
| 保留 | 向 ag-mem-02 请求重置槽位活跃时间戳为当前时间 |
| 删除 | 向 ag-mem-02 发起槽位清除请求，标注“用户主动删除” |
| 忽略 | 无操作，当前提醒周期结束，等待下一周期重新评估 |
| 超时（7天无响应） | 默认保留，向 ag-mem-02 请求重置活跃时间戳 |


## 核心处理逻辑

```
FUNCTION inactive_slot_reminder_main_loop():
    STATE_IDLE = IDLE
    STATE_SCAN = SCANNING
    STATE_AWAIT = AWAITING_USER
    STATE_PROCESS = PROCESSING_DECISION
    STATE_PAUSED = SYSTEM_PAUSED

    SET internal_state = STATE_IDLE
    初始化提醒记录表（槽位编号 → 提醒发送时间 + 用户响应状态）

    WHILE 系统运行中:
        // 第1步：紧急熔断
        IF 收到紧急熔断指令:
            SET internal_state = STATE_PAUSED
            CONTINUE
        ELSE IF 收到恢复指令 AND internal_state == STATE_PAUSED:
            SET internal_state = STATE_IDLE

        // 第2步：定时扫描触发
        IF 收到扫描触发信号 OR (距上次扫描 >= 24小时 AND internal_state == STATE_IDLE):
            SET internal_state = STATE_SCAN

            // 2a. 查询所有长期槽状态
            长期槽列表 = 向 ag-mem-02 查询所有长期槽状态()
            IF 长期槽列表 == None OR 长期槽列表为空:
                SET internal_state = STATE_IDLE
                CONTINUE

            // 2b. 遍历检测未活跃槽位
            未活跃槽位列表 = []
            FOR EACH 槽位 IN 长期槽列表:
                未活跃天数 = (NOW() - 槽位.最后活跃时间) / 86400  // 转为天数
                IF 未活跃天数 > 90:
                    未活跃槽位列表.append({
                        槽位编号: 槽位.槽位编号,
                        用户ID: 槽位.用户ID,
                        未活跃天数: 未活跃天数,
                        最后活跃时间: 槽位.最后活跃时间,
                        存储使用量: 槽位.存储使用量
                    })

            // 2c. 发送提醒通知
            IF 未活跃槽位列表 非空:
                FOR EACH 未活跃槽位 IN 未活跃槽位列表:
                    // 检查是否已发送过提醒且用户尚未响应
                    IF 提醒记录表.存在(未活跃槽位.槽位编号) AND 提醒记录.响应状态 == "等待中":
                        // 仍在等待上次提醒的响应，跳过重复发送
                        CONTINUE

                    向 ag-ecc-10 发送未活跃槽位提醒通知(
                        槽位编号=未活跃槽位.槽位编号,
                        用户ID=未活跃槽位.用户ID,
                        未活跃天数=未活跃槽位.未活跃天数,
                        建议选项=["保留", "删除", "忽略"]
                    )
                    提醒记录表.添加(未活跃槽位.槽位编号, 发送时间=NOW(), 响应状态="等待中")

                SET internal_state = STATE_AWAIT
            ELSE:
                SET internal_state = STATE_IDLE

            // 2d. 上报扫描结果
            向 ag-mem-01 上报扫描结果(
                扫描时间=NOW(),
                扫描槽位总数=长期槽列表.size,
                未活跃槽位数=未活跃槽位列表.size,
                已发送提醒数=未活跃槽位列表.size
            )

        // 第3步：等待用户响应
        IF internal_state == STATE_AWAIT:
            // 3a. 检查是否有用户决策响应
            用户响应 = 检查是否有 ag-ecc-10 转发的用户决策响应()
            IF 用户响应 != None:
                SET internal_state = STATE_PROCESS
                处理用户决策(用户响应)

            // 3b. 检查是否有超时未响应的提醒（7天）
            FOR EACH 提醒记录 IN 提醒记录表:
                IF 提醒记录.响应状态 == "等待中" AND (NOW() - 提醒记录.发送时间) > 7天:
                    // 超时默认保留
                    向 ag-mem-02 发送槽位活跃时间重置请求(
                        槽位编号=提醒记录.槽位编号,
                        新活跃时间戳=NOW(),
                        重置原因="未活跃提醒超时，默认保留"
                    )
                    提醒记录.响应状态 = "已超时默认保留"
                    向 ag-mem-51 发送提醒事件日志("超时默认保留", 提醒记录.槽位编号)

            // 3c. 检查是否所有提醒都已处理
            IF 提醒记录表中所有记录的响应状态 != "等待中":
                SET internal_state = STATE_IDLE

        // 第4步：处理用户决策
        IF internal_state == STATE_PROCESS:
            // 已在上面的步骤3a中处理，这里恢复状态
            // 检查是否还有待处理的响应
            IF 提醒记录表中无"等待中"的记录:
                SET internal_state = STATE_IDLE

        SLEEP 10秒  // 低频操作，10秒循环即可


FUNCTION 处理用户决策(用户响应):
    提醒记录 = 提醒记录表.查找(用户响应.槽位编号)
    IF 提醒记录 == None:
        RETURN  // 无效响应

    CASE 用户响应.用户决策 OF:
        "保留":
            向 ag-mem-02 发送槽位活跃时间重置请求(
                槽位编号=用户响应.槽位编号,
                新活跃时间戳=NOW(),
                重置原因="用户确认保留"
            )
            提醒记录.响应状态 = "已保留"
            向 ag-mem-51 发送提醒事件日志("用户保留", 用户响应.槽位编号)

        "删除":
            向 ag-mem-02 发送槽位清除请求(
                目标槽位编号=用户响应.槽位编号,
                清除原因="用户主动删除（未活跃提醒触发）",
                清除策略="长期槽删除（三次覆写）",
                用户确认标记=True
            )
            提醒记录.响应状态 = "已删除"
            向 ag-mem-51 发送提醒事件日志("用户删除", 用户响应.槽位编号)

        "忽略":
            提醒记录.响应状态 = "已忽略"
            向 ag-mem-51 发送提醒事件日志("用户忽略", 用户响应.槽位编号)
```


## 约束与异常处理

| 场景 | 处理方式 | 恢复条件 |
|------|----------|----------|
| 长期槽列表查询失败 | 跳过本次扫描，等待下一个扫描周期 | 查询恢复正常 |
| 提醒通知发送失败（ag-ecc-10 无响应） | 记录发送失败，等待下一周期重试 | ag-ecc-10 恢复正常 |
| 用户响应超时（7天无响应） | 默认保留，重置活跃时间戳，避免误删用户数据 | 用户后续手动操作 |
| 同一槽位重复提醒 | 通过提醒记录表去重，同一槽位在用户响应前不重复发送 | 用户响应后清除记录 |
| 紧急熔断 | 暂停扫描和提醒，保持当前提醒记录 | 紧急解除后恢复 |

## 安全边界

| 规则编号 | 内容 |
|:---:|------|
| S-01 | 未活跃提醒通知仅通过 ag-ecc-10 发送，不得包含用户的任何画像数据内容 |
| S-02 | 用户选择删除时，必须通过 ag-mem-02 发起正式的槽位清除流程，由 ag-mem-12 执行安全擦除 |
| S-03 | 超时默认保留策略优先于删除，避免因用户未看到通知而导致数据丢失 |
| S-04 | 提醒记录表仅存于内存，不持久化，系统重启后重新评估未活跃状态 |


## 接口校验用例

| 用例编号 | 前置条件 | 输入 | 预期输出 |
|----------|----------|------|----------|
| TC-M13-01 | `IDLE`，检测到1个槽位未活跃100天 | 扫描触发 + 长期槽列表含1个超90天未活跃槽位 | 发送提醒通知至 ag-ecc-10，进入 AWAITING_USER |
| TC-M13-02 | `AWAITING_USER`，用户选择保留 | 用户决策响应（槽位编号=SLOT-LONG-3，决策=保留） | 向 ag-mem-02 发送活跃时间重置请求，记录日志 |
| TC-M13-03 | `AWAITING_USER`，用户选择删除 | 用户决策响应（槽位编号=SLOT-LONG-3，决策=删除） | 向 ag-mem-02 发送槽位清除请求，记录日志 |
| TC-M13-04 | `AWAITING_USER`，7天无响应 | 提醒发送时间超过7天，无用户响应 | 默认保留，重置活跃时间戳，记录超时事件 |
| TC-M13-05 | `IDLE`，所有槽位均活跃 | 扫描触发 + 所有槽位最后活跃时间在90天内 | 不发送任何提醒，上报扫描结果（未活跃数=0） |
| TC-M13-06 | `IDLE`，同一槽位已有未处理提醒 | 扫描触发 + 该槽位已在提醒记录表中且状态为“等待中” | 跳过该槽位，不重复发送提醒 |
