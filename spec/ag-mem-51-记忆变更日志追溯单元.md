# ag-mem-51-记忆变更日志追溯单元 接口规格

---

## 基本信息

| 项 | 内容 |
|----|------|
| 模块编号 | ag-mem-51 |
| 模块名称 | 记忆变更日志追溯单元 |
| 所属分区 | 五、存储与系统运维 |
| 核心职责 | 全链路记录双漏斗记忆中枢中所有记忆操作事件的不可变审计日志。涵盖经验写入、晋升、遗忘、归档、导入导出、安全仲裁、警示标签变更等全部记忆生命周期操作。每条日志包含 UTC 时间戳、操作类型、操作模块、目标条目ID、变更前状态、变更后状态、操作原因摘要。日志存储采用追加写模式，禁止修改或删除已落盘的日志条目。存储周期 ≥ 3 年，为事故追溯、责任判定、合规审计提供完整的数据基础。支持按时间、操作类型、模块等多维度检索。不参与任何认知决策，仅负责日志的记录、存储与检索服务 |
| 依赖模块 | ag-mem-01 至 ag-mem-50（所有记忆模块均可向本模块推送变更日志事件）、ag-mem-42（冗余记忆删除与归档单元，推送删除/归档事件）、ag-mem-43（失败经验安全仲裁单元，推送仲裁事件）、ag-mem-50（记忆导入导出单元，推送导入导出事件） |
| 被依赖模块 | ag-mem-01（总控漏斗 F₀，查询历史日志供决策追溯）、ECC-12（资源调度模块，可通过 MemoryBus 查询事故相关日志）、系统管理接口、离线复盘系统 |


## 内部状态定义

| 状态 | 标识 | 含义 | 触发条件 |
|------|------|------|----------|
| 正常记录 | `NORMAL_LOGGING` | 正常接收并存储各模块推送的日志事件 | 系统初始化完成，存储介质就绪 |
| 存储空间不足 | `LOW_STORAGE` | 日志存储空间低于警戒阈值 | 剩余存储空间 < 15% |
| 存储故障 | `STORAGE_FAULT` | 存储介质写入失败或不可用 | 连续写入失败 ≥ 3 次 |
| 暂停服务 | `SYSTEM_PAUSED` | 系统紧急熔断 | 收到紧急熔断指令 |

## 日志条目结构与事件类型

### 一、日志条目结构

| 字段名称 | 数据类型 | 说明 |
|----------|----------|------|
| 日志条目ID | String | 唯一标识，格式：LOG-{序列号}-{时间戳} |
| 序列号 | UInt64 | 全局单调递增序列号，确保时序可追溯 |
| 事件类型 | Enum | 见下方事件类型枚举 |
| 来源模块编号 | String | 触发该操作的模块编号 |
| 目标条目ID | String | 被操作的经验条目ID |
| 变更前状态 | String | 操作前的状态描述（如：层级=L2, I值=0.65） |
| 变更后状态 | String | 操作后的状态描述（如：层级=L3, I值=0.72） |
| 操作原因摘要 | String | 人类可读的操作原因（如：满足晋升条件、I值低于遗忘阈值） |
| 关联数据引用 | Struct | 可选的关联数据引用（如：归并来源条目列表） |
| UTC 时间戳 | Float | 事件发生的精确 UTC 时间戳 |
| 校验签名 | String | 日志条目的 HMAC 签名，用于防篡改验证 |

### 二、日志事件类型枚举

| 事件类型 | 标识 | 说明 |
|----------|:---:|------|
| `EXPERIENCE_WRITE` | 经验写入 | 新经验写入 L1 临时层 |
| `EXPERIENCE_PROMOTE` | 经验晋升 | 经验从低层级晋升至高层级 |
| `EXPERIENCE_FORGET` | 经验遗忘 | 经验因遗忘判定被清除 |
| `EXPERIENCE_ARCHIVE` | 经验归档 | 经验被压缩归档至冷存储 |
| `EXPERIENCE_RESTORE` | 经验恢复 | 从冷存储恢复归档经验 |
| `EXPERIENCE_MERGE` | 经验归并 | L3 相似经验归并操作 |
| `SLOT_CREATE` | 画像槽创建 | 漏斗一用户画像槽创建 |
| `SLOT_DELETE` | 画像槽清除 | 漏斗一用户画像槽安全擦除 |
| `SAFETY_ARBITRATION` | 安全仲裁 | 失败经验安全仲裁结果 |
| `CAUTION_LABEL_CHANGE` | 警示标签变更 | 条目标签 CAUTION ↔ NORMAL 变更 |
| `L5_LOCK` | L5 锁定/解锁 | L5 核心层的写入权限变更 |
| `IMPORT` | 经验导入 | 外部经验包导入 |
| `EXPORT` | 经验导出 | 经验脱敏导出 |
| `CAPACITY_CLEANUP` | 容量清理 | 全局容量管控触发的清理操作 |
| `CONFIG_CHANGE` | 配置变更 | 三维权重系数等关键配置变更 |
| `SYSTEM_EVENT` | 系统事件 | 初始化、熔断、恢复等系统级事件 |


## 核心处理逻辑

```
FUNCTION change_log_tracer_main_loop():
    STATE_NORMAL = NORMAL_LOGGING
    STATE_LOW = LOW_STORAGE
    STATE_FAULT = STORAGE_FAULT
    STATE_PAUSED = SYSTEM_PAUSED

    SET internal_state = STATE_NORMAL
    初始化日志存储（追加写文件或数据库）
    初始化全局序列号 = 0
    初始化内存缓冲队列（用于存储故障时的暂存）
    写入失败计数器 = 0

    WHILE 系统运行中:
        // 第1步：紧急熔断
        IF 收到紧急熔断指令:
            SET internal_state = STATE_PAUSED
            尝试强制刷写内存缓冲队列到存储
            CONTINUE
        ELSE IF 收到恢复指令 AND internal_state == STATE_PAUSED:
            SET internal_state = STATE_NORMAL

        // 第2步：接收并记录日志事件
        日志事件 = 获取下一个待记录事件()
        WHILE 日志事件 != None:
            全局序列号++
            当前时间戳 = NOW()

            // 2a. 构建日志条目
            日志条目 = 构建日志条目(
                日志条目ID="LOG-" + 全局序列号 + "-" + 当前时间戳,
                序列号=全局序列号,
                事件类型=日志事件.事件类型,
                来源模块编号=日志事件.来源模块编号,
                目标条目ID=日志事件.目标条目ID,
                变更前状态=日志事件.变更前状态,
                变更后状态=日志事件.变更后状态,
                操作原因摘要=日志事件.操作原因摘要,
                关联数据引用=日志事件.关联数据引用,
                UTC时间戳=当前时间戳,
                校验签名=HMAC签名(上述字段)
            )

            // 2b. 持久化写入
            IF internal_state == STORAGE_FAULT:
                暂存至内存缓冲队列()
                IF 内存缓冲队列.size > 5000:
                    丢弃最旧的日志条目（记录告警）
            ELSE:
                写入结果 = 日志存储.追加写入(日志条目)
                IF 写入结果.失败:
                    写入失败计数器++
                    IF 写入失败计数器 >= 3:
                        SET internal_state = STATE_FAULT
                        向 ag-mem-01 发送存储告警("存储故障", "连续写入失败3次")
                        暂存至内存缓冲队列()
                ELSE:
                    写入失败计数器 = 0
                    // 尝试回写缓冲队列中的数据
                    回写内存缓冲队列()

            // 2c. 向事件来源返回确认（仅对标记为需要确认的事件）
            IF 日志事件.需要确认:
                向来源模块返回日志写入确认(日志条目.日志条目ID, 写入结果)

            日志事件 = 获取下一个待记录事件()

        // 第3步：存储空间管理
        IF internal_state != STORAGE_FAULT:
            存储状态 = 查询日志存储状态()
            剩余比例 = 存储状态.剩余容量 / 存储状态.总容量

            IF 剩余比例 < 10%:
                SET internal_state = STATE_LOW
                触发旧日志清理(保留最近 3 年数据)
                向 ag-mem-01 发送存储告警("空间不足", "日志存储空间低于10%")
            ELSE IF 剩余比例 < 5%:
                触发强制清理(保留最近 2 年数据)
                向 ag-mem-01 发送存储告警("空间严重不足", "日志存储空间低于5%，已触发强制清理")

        // 第4步：响应日志查询请求
        IF 收到日志查询请求:
            查询开始时间 = NOW()
            匹配日志列表 = 日志存储.按条件检索(
                时间范围=请求.时间范围,
                模块ID=请求.模块ID,
                事件类型=请求.事件类型,
                条目ID=请求.条目ID,
                关键字=请求.关键字
            )
            匹配总数 = 匹配日志列表.size
            匹配日志列表 = 截断(匹配日志列表, 请求.最大返回条数)
            查询耗时 = NOW() - 查询开始时间

            向请求来源返回日志查询结果(
                查询条件=请求,
                匹配日志列表=匹配日志列表,
                匹配总数=匹配总数,
                查询耗时=查询耗时,
                是否完整=匹配日志列表.size == 匹配总数
            )

        // 第5步：响应日志导出请求
        IF 收到日志导出请求:
            // 与查询类似，但输出为可下载的日志包
            导出日志列表 = 日志存储.按条件检索(请求.导出范围)
            日志导出包 = 构建日志导出包(导出日志列表, 请求.导出格式)
            日志导出包.校验签名 = SHA256(日志导出包.日志条目列表)
            向请求来源返回日志导出包

        // 第6步：周期性状态上报
        IF 距上次状态上报 >= 60秒:
            存储状态 = 查询日志存储状态()
            向 ag-mem-01 和 ag-mem-03 上报存储状态(
                当前状态=internal_state,
                总容量=存储状态.总容量,
                已用容量=存储状态.已用容量,
                剩余容量=存储状态.剩余容量,
                最早日志时间=存储状态.最早日志时间,
                最新日志时间=存储状态.最新日志时间,
                写入速率=计算写入速率()
            )

        SLEEP 10ms
```


## 约束与异常处理

| 场景 | 处理方式 | 恢复条件 |
|------|----------|----------|
| 日志存储写入连续失败 ≥ 3 次 | 进入 STORAGE_FAULT，切换至内存缓冲，上报告警 | 存储恢复正常且后续写入成功 |
| 内存缓冲队列溢出（>5000条） | 丢弃最旧的日志条目，记录告警（优先保留安全事件类日志） | 存储恢复后回写缓冲数据 |
| 日志存储空间低于 10% | 触发旧日志清理（保留最近 3 年），上报告警 | 空间释放至 20% 以上 |
| 日志存储空间低于 5% | 触发强制清理（保留最近 2 年），上报告警 | 空间释放至 15% 以上 |
| 查询请求超时（>500ms） | 返回部分结果并标记“查询未完成” | — |
| 紧急熔断 | 强制刷写内存缓冲至存储（若可用），暂停接收新事件 | 紧急解除后恢复 |
| 日志条目 HMAC 签名生成失败 | 使用空签名占位，标记“签名异常”，继续记录（不中断日志链路） | 签名密钥恢复 |

## 安全边界

| 规则编号 | 内容 |
|:---:|------|
| L-01 | 日志存储必须采用追加写模式（Append-Only），禁止任何模块修改或删除已落盘的日志条目 |
| L-02 | 每条日志条目必须包含全局单调递增序列号与 UTC 时间戳，确保时序不可篡改 |
| L-03 | 事故相关的关键日志（安全仲裁、L5 锁定、配置变更）在存储空间不足时优先保留，不得被自动清理 |
| L-04 | 日志存储周期 ≥ 3 年，超期日志在空间不足时优先清理，但需在清理前发送告警通知 |
| L-05 | 日志 HMAC 签名用于内部完整性校验，发现签名不匹配的日志条目时应上报告警但不中断服务 |
